When most people say "agentic system," they mean a chain of LLM calls with tool use. That demos well. It does not survive production.

The systems I have shipped - Follicle Labeller at Kollestee UK Global, and CouncilAgent at the Microsoft Embrace Midlands Hackathon - work because of three constraints I keep coming back to.

## 1. The agent should fail loudly, not silently

Every tool call has a cost: a model token, a database write, a clinician's attention. The expensive failure mode is "the agent confidently does the wrong thing." The cheap failure mode is "the agent says it cannot proceed."

I bias every decision toward the cheap failure. That means strict output schemas (Zod or Pydantic), tool calls that throw on ambiguity, and human-in-the-loop checkpoints wherever the action is irreversible.

## 2. Retrieval is more important than generation

When a generative system gives a wrong answer, 80% of the time the cause is bad retrieval, not bad generation. The model was given the wrong context, or no context.

So I spend my budget on retrieval: chunking, embedding strategy, reranking, hybrid search. For my Board of Directors RAG, the five-stage retrieval pipeline (BM25 sparse → vector → reranker → persona filter → recency boost) does more for answer quality than any prompt tweak ever did.

## 3. Evals before features

Every agent I build now has an eval set before I write a single feature. Twenty to fifty examples, hand-labeled, run on every PR. Without this, "did I break something" becomes a feeling instead of a number.

The team at Anthropic talks about this constantly and they are right. If you cannot quantify a regression, you will ship one.

## What I would tell my past self

If I could go back, I would build the eval harness first, the retrieval pipeline second, and only then touch the agent loop. The order matters. Agent loops are the most visible part and the least valuable part to start with - the failure modes are downstream.
